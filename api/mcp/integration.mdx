## Client Integration

### Connecting to the Server

The PairOfCleats MCP server communicates over stdio using JSON-RPC 2.0. Launch the server as a subprocess:

```javascript
import { spawn } from 'child_process';

const server = spawn('node', ['tools/mcp/server.js', '--mcp-mode', 'sdk'], {
  stdio: ['pipe', 'pipe', 'inherit'],
  cwd: '/path/to/pairofcleats'
});

// Read from server.stdout
// Write to server.stdin
```

### Initialize Handshake

After launching the server, send an `initialize` request:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2024-11-05",
    "clientInfo": {
      "name": "my-mcp-client",
      "version": "1.0.0"
    },
    "capabilities": {}
  }
}
```

The server responds with:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2024-11-05",
    "serverInfo": {
      "name": "pairofcleats-mcp",
      "version": "0.0.1"
    },
    "capabilities": {
      "experimental": {
        "pairofcleats": {
          "schemaVersion": "1.2.0",
          "toolVersion": "0.0.1"
        }
      }
    },
    "schemaVersion": "1.2.0",
    "toolVersion": "0.0.1"
  }
}
```

<Note>
Store the `schemaVersion` to handle tool schema compatibility in your client.
</Note>

### Listing Available Tools

Request the list of available tools:

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/list"
}
```

Response includes all tool definitions with input schemas:

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "tools": [
      {
        "name": "search",
        "description": "Run a search query against the repo index.",
        "inputSchema": {
          "type": "object",
          "properties": {
            "query": { "type": "string" },
            "mode": { "type": "string", "enum": ["both", "code", "prose"] },
            "top": { "type": "number" }
          },
          "required": ["query"]
        }
      }
    ]
  }
}
```

### Calling Tools

Invoke a tool using `tools/call`:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "search",
    "arguments": {
      "query": "authentication middleware",
      "mode": "code",
      "top": 5
    }
  }
}
```

Successful response:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{\"results\":[...],\"totalHits\":42}"
      }
    ]
  }
}
```

<Note>
Tool results are returned as JSON strings in `result.content[0].text`. Parse this string to access the structured data.
</Note>

## Error Handling

### Error Response Format

When a tool call fails, the server returns:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "isError": true,
    "content": [
      {
        "type": "text",
        "text": "{\"code\":\"INVALID_REQUEST\",\"namespaceCode\":\"poc.invalid_request\",\"message\":\"Query is required.\"}"
      }
    ]
  }
}
```

### Protocol-Level Errors

JSON-RPC protocol errors use standard error objects:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "error": {
    "code": -32001,
    "message": "Request queue is full",
    "data": {
      "code": "QUEUE_OVERLOADED",
      "namespaceCode": "poc.queue_overloaded"
    }
  }
}
```

### Error Codes

All errors include a canonical `code` field:

<ResponseField name="INVALID_REQUEST" type="string">
  Arguments are missing, invalid, or an unsupported option was provided
</ResponseField>

<ResponseField name="NO_INDEX" type="string">
  Required index artifacts are missing. Run `build_index` first.
</ResponseField>

<ResponseField name="TOOL_TIMEOUT" type="string">
  Tool execution exceeded its configured timeout
</ResponseField>

<ResponseField name="CAPABILITY_MISSING" type="string">
  Required optional capability or backend is not available
</ResponseField>

<ResponseField name="CANCELLED" type="string">
  Request was cancelled before completion
</ResponseField>

<ResponseField name="QUEUE_OVERLOADED" type="string">
  Server queue rejected the request due to load. Retry later or increase `mcp.queueMax`.
</ResponseField>

<ResponseField name="NOT_FOUND" type="string">
  Requested tool or resource was not found
</ResponseField>

<ResponseField name="UNAUTHORIZED" type="string">
  Missing or invalid authentication
</ResponseField>

<ResponseField name="FORBIDDEN" type="string">
  Authenticated but not permitted to perform the action
</ResponseField>

<ResponseField name="DOWNLOAD_VERIFY_FAILED" type="string">
  Download failed integrity verification
</ResponseField>

<ResponseField name="ARCHIVE_UNSAFE" type="string">
  Archive contents are unsafe (path traversal or policy violation)
</ResponseField>

<ResponseField name="ARCHIVE_TOO_LARGE" type="string">
  Archive exceeds size limits
</ResponseField>

<ResponseField name="INTERNAL" type="string">
  Unexpected failure not covered by other error codes
</ResponseField>

### Error Payload Fields

Error payloads may include:

```typescript
interface ErrorPayload {
  code: string;              // Canonical error code
  namespaceCode: string;     // Namespaced code (poc.error_code)
  message: string;           // Human-readable message
  stderr?: string;           // stderr output if available
  stdout?: string;           // stdout output if available
  timeoutMs?: number;        // Timeout value for TOOL_TIMEOUT
  exitCode?: number;         // Process exit code if applicable
  hint?: string;             // Suggestion for resolution
}
```

### Handling Common Errors

#### Missing Index

```javascript
if (error.code === 'NO_INDEX') {
  // Build the index first
  await callTool('build_index', { mode: 'all' });
  // Retry the original request
}
```

#### Queue Overload

```javascript
if (error.code === 'QUEUE_OVERLOADED') {
  // Implement exponential backoff
  await delay(1000 * Math.pow(2, retryCount));
  // Retry the request
}
```

#### Tool Timeout

```javascript
if (error.code === 'TOOL_TIMEOUT') {
  // Increase timeout in config or split work
  console.error(`Tool timed out after ${error.timeoutMs}ms`);
  console.error(`Hint: ${error.hint}`);
}
```

## Progress Notifications

Long-running tools emit `notifications/progress` to report status:

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/progress",
  "params": {
    "progressToken": "build_index_123",
    "progress": {
      "id": 0,
      "tool": "build_index",
      "message": "Building code index (working)",
      "stream": "stdout",
      "phase": "progress",
      "ts": "2024-03-04T12:34:56.789Z"
    }
  }
}
```

### Progress Phases

<ParamField path="phase" type="string">
  - `start`: Tool execution has started
  - `progress`: Ongoing progress update
  - `done`: Tool execution completed successfully
</ParamField>

### Handling Progress

```javascript
function handleNotification(notification) {
  if (notification.method === 'notifications/progress') {
    const { tool, message, phase } = notification.params.progress;
    
    if (phase === 'start') {
      console.log(`[${tool}] Starting: ${message}`);
    } else if (phase === 'progress') {
      console.log(`[${tool}] ${message}`);
    } else if (phase === 'done') {
      console.log(`[${tool}] Complete: ${message}`);
    }
  }
}

// Listen for notifications on stdout
server.stdout.on('data', (chunk) => {
  const messages = parseJsonRpc(chunk);
  messages.forEach(msg => {
    if (msg.method) {
      handleNotification(msg);
    }
  });
});
```

<Note>
Progress notification `id=0` is valid. Only `null` or `undefined` indicates a missing ID.
</Note>

## Request Cancellation

Cancel in-flight requests using `$/cancelRequest`:

```json
{
  "jsonrpc": "2.0",
  "method": "$/cancelRequest",
  "params": { "id": 3 }
}
```

The cancelled request receives an error response:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "isError": true,
    "content": [
      {
        "type": "text",
        "text": "{\"code\":\"CANCELLED\",\"message\":\"Request cancelled.\"}"
      }
    ]
  }
}
```

### Implementing Cancellation

```javascript
class McpClient {
  constructor() {
    this.pendingRequests = new Map();
  }

  async callTool(name, args, options = {}) {
    const id = this.nextId++;
    const controller = new AbortController();
    
    this.pendingRequests.set(id, controller);
    
    // Set up timeout cancellation
    if (options.timeout) {
      setTimeout(() => this.cancel(id), options.timeout);
    }
    
    try {
      return await this.sendRequest({
        jsonrpc: '2.0',
        id,
        method: 'tools/call',
        params: { name, arguments: args }
      });
    } finally {
      this.pendingRequests.delete(id);
    }
  }

  cancel(id) {
    this.sendNotification({
      jsonrpc: '2.0',
      method: '$/cancelRequest',
      params: { id }
    });
    this.pendingRequests.get(id)?.abort();
  }
}
```

## Transport Framing

### Legacy Mode (Content-Length)

Messages use Content-Length headers:

```
Content-Length: 123\r\n
\r\n
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}
```

### SDK Mode (Newline-Delimited)

Messages are newline-delimited JSON:

```
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}\n
{"jsonrpc":"2.0","id":2,"method":"tools/list"}\n
```

<Note>
The server automatically handles framing based on the selected `--mcp-mode`. Clients must match the framing format.
</Note>

## Best Practices

### Connection Management

1. Always send `initialize` before any other requests
2. Store `schemaVersion` to handle compatibility
3. Implement reconnection logic for subprocess failures
4. Clean up stdio pipes on disconnect

### Request Handling

1. Parse JSON from `result.content[0].text` for tool results
2. Check `isError` flag before parsing results
3. Implement retry logic for transient errors (`QUEUE_OVERLOADED`)
4. Set appropriate timeouts for long-running operations

### Progress Monitoring

1. Listen for `notifications/progress` on all long-running tools
2. Display progress to users for operations like `build_index`, `download_models`
3. Track `phase` to determine operation lifecycle
4. Log progress messages for debugging

### Error Recovery

1. Handle `NO_INDEX` by running `build_index` first
2. Implement exponential backoff for `QUEUE_OVERLOADED`
3. Respect `hint` fields in error payloads
4. Log `stderr` and `stdout` for debugging `INTERNAL` errors

### Performance

1. Reuse single server instance across multiple requests
2. Configure `mcp.queueMax` based on expected load
3. Adjust `mcp.maxBufferBytes` for large result sets
4. Use `output: "compact"` for search queries when possible

## Example Client

```javascript
import { spawn } from 'child_process';
import { createInterface } from 'readline';

class PairOfCleatsMcpClient {
  constructor(serverPath) {
    this.serverPath = serverPath;
    this.nextId = 1;
    this.pendingRequests = new Map();
    this.connected = false;
  }

  async connect() {
    this.server = spawn('node', [this.serverPath, '--mcp-mode', 'sdk'], {
      stdio: ['pipe', 'pipe', 'inherit']
    });

    this.readline = createInterface({
      input: this.server.stdout,
      output: null
    });

    this.readline.on('line', (line) => this.handleMessage(JSON.parse(line)));

    await this.initialize();
    this.connected = true;
  }

  async initialize() {
    const response = await this.sendRequest({
      jsonrpc: '2.0',
      id: this.nextId++,
      method: 'initialize',
      params: {
        protocolVersion: '2024-11-05',
        clientInfo: { name: 'example-client', version: '1.0.0' },
        capabilities: {}
      }
    });
    this.schemaVersion = response.result.schemaVersion;
    return response;
  }

  handleMessage(message) {
    if (message.method === 'notifications/progress') {
      this.onProgress?.(message.params.progress);
    } else if (message.id) {
      const resolve = this.pendingRequests.get(message.id);
      if (resolve) {
        this.pendingRequests.delete(message.id);
        resolve(message);
      }
    }
  }

  sendRequest(request) {
    return new Promise((resolve, reject) => {
      this.pendingRequests.set(request.id, resolve);
      this.server.stdin.write(JSON.stringify(request) + '\n');
      
      // Timeout after 2 minutes
      setTimeout(() => {
        if (this.pendingRequests.has(request.id)) {
          this.pendingRequests.delete(request.id);
          reject(new Error('Request timeout'));
        }
      }, 120000);
    });
  }

  async callTool(name, args) {
    const response = await this.sendRequest({
      jsonrpc: '2.0',
      id: this.nextId++,
      method: 'tools/call',
      params: { name, arguments: args }
    });

    if (response.error) {
      throw new Error(response.error.message);
    }

    const result = response.result;
    if (result.isError) {
      const error = JSON.parse(result.content[0].text);
      throw new Error(`${error.code}: ${error.message}`);
    }

    return JSON.parse(result.content[0].text);
  }

  disconnect() {
    this.server.kill();
    this.connected = false;
  }
}

// Usage
const client = new PairOfCleatsMcpClient('tools/mcp/server.js');

client.onProgress = (progress) => {
  console.log(`[${progress.tool}] ${progress.message}`);
};

await client.connect();

try {
  const status = await client.callTool('index_status', {});
  console.log('Index status:', status);

  const results = await client.callTool('search', {
    query: 'authentication',
    mode: 'code',
    top: 10
  });
  console.log('Search results:', results);
} catch (error) {
  console.error('Error:', error);
} finally {
  client.disconnect();
}
```

## Schema Validation

The server validates tool inputs against the schema defined in `docs/contracts/mcp-tools.schema.json`. Clients should:

1. Fetch the schema from the canonical location
2. Validate inputs before sending to reduce round-trips
3. Handle schema version mismatches gracefully
4. Update client when server reports new `schemaVersion`

```javascript
import Ajv from 'ajv';

const ajv = new Ajv();
const toolSchema = await fetch('/path/to/mcp-tools.schema.json').then(r => r.json());

function validateToolCall(name, args) {
  const tool = toolSchema.tools.find(t => t.name === name);
  if (!tool) {
    throw new Error(`Unknown tool: ${name}`);
  }
  
  const validate = ajv.compile(tool.inputSchema);
  if (!validate(args)) {
    throw new Error(`Invalid arguments: ${JSON.stringify(validate.errors)}`);
  }
}
```
